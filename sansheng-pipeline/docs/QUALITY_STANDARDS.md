**版本**: v1.0
**日期**: 2026-03-11
**编撰**: 礼部

## 一、总体原则

### 1.1 渐进式检查

门下省采用**渐进式质量检查**机制，每次封驳只提出**一个维度**的改进要求，而不是一次性列出所有问题。

**理念**：
- 降低中书省单次修改负担
- 避免信息过载导致遗漏
- 便于追踪修改质量
- 符合迭代开发思想

### 1.2 封驳上限

- **最大封驳次数**：5 次
- **升级条件**：第 5 次封驳后仍不合格 → 自动升级司礼监裁决
- **升级流程**：司礼监整理双方争议点，用户做最终裁决

---

## 二、基础检查（每次必查）

以下检查在每次审议时都必须执行，任何一项不通过即封驳：

### 2.1 长度检查
- **标准**：方案总长度 ≥ 100 字
- **封驳理由**："方案过于简略（长度 {len} 字），应不少于 100 字"
- **修改建议**：扩充内容描述，补充必要细节

### 2.2 结构检查
- **标准**：必须包含"步骤"或"方案"关键词
- **封驳理由**："方案缺少关键结构，应包含'步骤'或'方案'章节"
- **修改建议**：使用标准章节标题（如"实施步骤""执行方案"）

### 2.3 目标检查
- **标准**：必须包含"目标"或"背景"章节
- **封驳理由**："方案缺少目标说明，应包含'目标'或'背景'章节"
- **修改建议**：明确任务目标和背景信息

### 2.4 可执行性检查
- **标准**：有效内容行（非标题行）≥ 5 行
- **封驳理由**："方案内容过于简单，缺少具体描述（有效内容行少于 5 行）"
- **修改建议**：补充具体操作描述，避免纯标题式方案

### 2.5 完整性检查
- **标准**：章节数量足够（`###` ≥ 2 或 `##` ≥ 3）
- **封驳理由**："方案结构不完整，应包含多个章节"
- **修改建议**：拆分为多个章节，便于阅读和审查

## 三、第 0 次审议：风险评估

### 3.1 检查内容

**必须包含**：
- "风险"或"应对"关键词
- 至少 3 种风险类型

**标准风险类型**：
- 技术风险：新技术栈、兼容性、性能瓶颈
- 集成风险：与现有系统对接、数据迁移
- 时间风险：关键路径、资源冲突
- 人力风险：团队能力、协作依赖
- 外部风险：第三方服务、政策变化

### 3.2 合格示例

```markdown
### 四、风险与应对

- **技术风险**：新引入 Redis 作为缓存层，团队需要学习 Redis 最佳实践
  - **应对**：预留 2 天学习时间，参考官方文档和最佳实践

- **集成风险**：与现有数据库连接池需要兼容，可能存在连接泄漏
  - **应对**：开发阶段增加连接监控，压力测试验证

- **时间风险**：如果缓存方案失败，回退到原方案可能延误交付
  - **应对**：准备回滚脚本，保留原代码分支
```

### 3.3 封驳理由

- "方案缺少风险评估章节，应包含'风险与应对'"
- "风险评估过于简单，应至少列出 3 种风险类型"

---

## 四、第 1 次审议：验收标准

### 4.1 检查内容

**必须包含**：
- "验收"或"标准"关键词
- 功能验收、性能基准、代码质量

**标准验收维度**：
- 功能完整性：所有需求点通过测试
- 性能指标：满足 SLA 要求（具体指标）
- 代码质量：通过 code review、linting、测试覆盖率
- 安全合规：权限控制、数据保护
- 文档完整：API 文档、使用说明

### 4.2 合格示例

```markdown
### 五、验收标准

- **功能完整性**：
  - 用户可以通过手机号 + 验证码登录
  - 验证码 5 分钟内有效
  - 验证码错误 3 次后锁定账号 30 分钟

- **性能指标**：
  - 登录接口响应时间 P95 < 200ms
  - 验证码发送成功率 > 99%
  - 支持 1000 QPS 并发登录

- **代码质量**：
  - 单元测试覆盖率 > 80%
  - 通过 ESLint 检查，无 error 级别警告
  - Code review 通过，无 blocker 问题
```

### 4.3 封驳理由

- "方案缺少验收标准章节，应包含'验收标准'"
- "验收标准不具体，应明确功能、性能、质量的具体指标"

---

## 五、第 2 次审议：技术细节

### 5.1 检查内容

**必须包含**：
- "技术细节"或"技术栈"或"架构设计"关键词
- 技术栈选型、架构设计、实施细节
- **禁止"待补充"标记**

**标准技术细节**：
- 技术栈选型：语言、框架、数据库、中间件（需给出版本号）
- 架构设计：模块划分、数据流、通信方式
- 实施细节：关键算法、数据结构、配置方式

### 5.2 合格示例

```markdown
### 六、技术细节说明

- **技术栈选型**：
  - 后端：Node.js 18+, Express 4.18, Redis 7.0
  - 数据库：PostgreSQL 15（用户表）, Redis（验证码缓存）
  - 短信网关：阿里云短信服务 SDK v2.0

- **架构设计**：
  - 三层架构：Controller → Service → Repository
  - 验证码生成：6 位随机数字，Redis 存储，TTL 300s
  - 限流策略：单手机号 1 分钟最多发送 1 次

- **实施细节**：
  - 验证码生成算法：Math.random() * 1000000，补零到 6 位
  - Redis Key 格式：`sms:verify:{phone}:{timestamp}`
  - 短信模板 ID：SMS_123456789（需在阿里云后台配置）
```

### 5.3 封驳理由

- "方案缺少技术细节说明，应包含'技术细节说明'章节，详细描述技术栈选型、架构设计和实施细节"
- "技术细节说明内容不完整，存在'待补充'标记，请提供具体的技术方案"

---

## 六、第 3 次审议：性能指标

### 6.1 检查内容

**必须包含**：
- "性能"或"响应时间"或"QPS"或"吞吐量"关键词
- 响应时间、吞吐量、资源消耗、并发处理

**标准性能指标**：
- 响应时间：P50/P95/P99 百分位
- 吞吐量：QPS（每秒请求数）
- 资源消耗：CPU 使用率、内存占用
- 并发处理：最大并发连接数、并发用户数

### 6.2 合格示例

```markdown
### 七、性能指标

- **响应时间**：
  - P50: < 50ms
  - P95: < 200ms
  - P99: < 500ms

- **吞吐量**：
  - 正常负载：1000 QPS
  - 峰值负载：2000 QPS（双 11、活动期间）

- **资源消耗**：
  - CPU 使用率：< 70%（正常负载）
  - 内存占用：< 2GB（单实例）
  - Redis 连接数：< 100（连接池上限）

- **并发处理**：
  - 支持 >= 100 并发请求
  - 数据库连接池：20 连接
  - 无阻塞 I/O（异步处理）
```

### 6.3 封驳理由

- "方案缺少性能指标章节，应包含响应时间、吞吐量、资源消耗等性能指标"
- "性能指标不够具体，应给出 P95/P99 响应时间和 QPS 目标"

---

## 七、第 4 次审议：安全审计

### 7.1 检查内容

**必须包含**：
- "安全"或"权限"或"加密"或"审计日志"关键词
- 权限控制、数据加密、审计日志、防护措施

**标准安全措施**：
- 权限控制：RBAC（基于角色的访问控制）
- 数据加密：传输加密（HTTPS/TLS）、存储加密（敏感字段）
- 审计日志：关键操作记录（谁、何时、做了什么）
- 防护措施：SQL 注入、XSS、CSRF、DDoS

### 7.2 合格示例

```markdown
### 八、安全审计

- **权限控制**：
  - 采用 RBAC 模型，角色分为：admin, user, guest
  - 接口级权限：使用 Express 中间件验证用户角色
  - 数据级权限：用户只能查看自己的订单

- **数据加密**：
  - 传输加密：全站 HTTPS（TLS 1.3）
  - 存储加密：用户密码使用 bcrypt 加密（cost=10）
  - 敏感字段：身份证号、手机号使用 AES-256 加密

- **审计日志**：
  - 记录内容：用户登录、密码修改、敏感数据访问
  - 存储格式：JSONL，按日期分文件
  - 保留周期：180 天

- **防护措施**：
  - SQL 注入：使用 ORM 参数化查询，禁止拼接 SQL
  - XSS：前端输出使用 DOMPurify 过滤
  - CSRF：使用 csurf 中间件生成 token
  - 限流：单 IP 每分钟最多 100 次请求
```

### 7.3 封驳理由

- "方案缺少安全审计章节，应包含权限控制、数据加密、审计日志等安全措施"
- "安全审计不够全面，应至少覆盖权限控制、数据加密、审计日志、常见漏洞防护"

## 八、封驳格式规范

### 8.1 标准格式

```markdown
【封驳理由】
1. [问题描述]（严重程度：major/minor）
   - 影响：[具体影响]
   - 建议：[具体修改建议]

2. [问题描述]（严重程度：major/minor）
   - 影响：[具体影响]
   - 建议：[具体修改建议]

【审议维度】
- 可行性：pass/fail
- 完整性：pass/fail
- 风险控制：pass/fail
- 清晰度：pass/fail
- 针对性：pass/fail（仅修改版本）
```

### 8.2 严重程度定义

**major（严重）**：
- 缺少关键章节（风险评估、验收标准、技术细节）
- 方案不可执行（步骤缺失、逻辑矛盾）
- 存在重大风险且无应对措施
- 性能/安全指标缺失或不合理

**minor（轻微）**：
- 格式不规范（标题层级混乱）
- 表述不清晰（存在歧义）
- 细节不够完善（示例代码缺失）
- 文档格式问题（错别字、排版）

## 九、准奏条件

满足以下所有条件时，门下省应予以准奏：

1. ✅ 通过所有基础检查（长度、结构、目标、可执行性、完整性）
2. ✅ 包含风险评估（至少 3 种风险类型 + 应对措施）
3. ✅ 包含验收标准（功能、性能、质量指标明确）
4. ✅ 包含技术细节（技术栈、架构、实施细节清晰，无"待补充"）
5. ✅ 包含性能指标（响应时间、吞吐量、资源消耗）
6. ✅ 包含安全审计（权限、加密、日志、防护措施）
7. ✅ 无致命缺陷（逻辑矛盾、重大风险未覆盖）

**注意**：
- 轻微问题（minor）不阻止准奏，可在执行阶段完善
- 如果方案已达到"80 分标准"，即使不完美也应准奏
- 第 3 次修改后，如果问题不致命，应倾向于准奏

## 十、升级裁决

### 10.1 触发条件

- 连续封驳 **5 次**，仍有致命问题
- 中书省与门下省对某个技术选型存在根本性分歧（如 30 天 vs 7 天兼容期）

### 10.2 裁决流程

1. 门下省整理争议点清单
2. 司礼监整理双方观点（中书省观点 vs 门下省观点）
3. 司礼监提供裁决选项（A: 采纳中书省 / B: 采纳门下省 / C: 折中方案）
4. 用户做最终裁决
5. 中书省按裁决结果修改方案

### 10.3 争议点模板

```markdown
# 任务 {task_id} 裁决申请

## 争议概况
- 任务：{task_title}
- 封驳次数：5
- 核心分歧：{core_conflict}

## 双方观点

### 门下省观点
1. {menxia_point_1}
   - 理由：{reason}
   - 风险：{risk}

### 中书省观点
1. {zhongshu_point_1}
   - 理由：{reason}
   - 依据：{evidence}

## 裁决选项

**选项 A：采纳门下省意见**
- 动作：{action}
- 优点：{pros}
- 缺点：{cons}

**选项 B：采纳中书省方案**
- 动作：{action}
- 优点：{pros}
- 缺点：{cons}

**选项 C：折中方案**
- 动作：{action}
- 优点：{pros}

## 建议
{recommendation}

---
请用户裁决：选择 A / B / C，或给出新方向
```

## 十一、常见问题

### Q1: 为什么采用渐进式检查？

**A**: 一次性抛出 5 个维度的问题会导致：
- 中书省信息过载，容易遗漏某些修改
- 难以追踪修改质量（不知道每次改了哪些）
- 增加沟通成本（需要多次确认）

渐进式检查让中书省每次只关注一个维度，提高修改质量。

### Q2: 5 次封驳够用吗？

**A**: 根据实际经验：
- 简单任务：1-2 次封驳即可准奏
- 中等任务：3-4 次封驳
- 复杂任务：5 次封驳 + 可能升级裁决

如果 5 次仍不够，说明任务本身定义不清晰，需要用户介入。

### Q3: 门下省可以跳过某些维度吗？

**A**: 不可以。渐进式检查要求按顺序检查：
- 第 0 次必须检查风险评估
- 第 1 次必须检查验收标准
- 第 2 次必须检查技术细节
- ...

跳过会导致方案质量下降。

### Q4: 如果中书省在第 1 次就补全了所有 5 个维度怎么办？

**A**: 门下省仍然按渐进式检查：
- 第 0 次只检查风险评估，其他维度暂不审查
- 如果风险评估合格，进入第 1 次审议
- 如果所有维度都合格，最多 5 次审议后准奏

不会因为"已经全补了"就跳过审议轮次。
