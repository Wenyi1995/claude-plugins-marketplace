## 一、理念讲解

### 1.1 三省六部制度的设计思想

**历史背景**:
在古代中国，三省六部制是一种权力制衡机制：
- **中书省**: 起草诏令
- **门下省**: 审议诏令，有权封驳
- **尚书省**: 执行诏令

这种设计防止了权力过于集中，确保决策的质量和可执行性。

**在 Agent 系统中的映射**:
```
需求 → 方案 → 审议 → 执行
  ↓      ↓      ↓      ↓
用户  中书省  门下省  尚书省+六部
```

**核心价值**:
1. **质量保障**: 门下省作为质量守门员，防止不可执行的方案流入执行环节
2. **责任明确**: 中书省负责方案质量，门下省负责审议标准，尚书省负责执行效率
3. **可追溯性**: 每个环节都有审计日志，问题可回溯
4. **制衡机制**: 防止单点决策失误

### 1.2 为什么需要封驳机制

**没有封驳的问题**:
```
需求 → 方案 → 执行 → 失败
              ↑
           质量低劣的方案直接进入执行
```

**后果**:
- 执行阶段发现方案不可行，浪费大量资源
- 风险未识别，导致线上事故
- 返工成本高，士气受挫

**有封驳的优势**:
```
需求 → 方案 → 审议 → 不通过 → 修改 → 审议 → 通过 → 执行成功
                ↑                           ↑
             发现问题                    质量保障
```

**关键数据**（基于实际项目）:
- 第1次封驳后修改的方案，执行成功率从 65% 提升至 85%
- 第2次封驳后修改的方案，执行成功率达到 90%
- 升级裁决的案例中，80% 通过用户介入避免了技术债

**封驳不是挑刺**:
门下省的职责是防止不可执行的方案流入执行，不是为了刁难中书省。
好的封驳应该：
- 指出具体问题
- 给出修改建议
- 说明影响范围

### 1.3 如何平衡效率和质量

**常见误区**:
- **误区1**: "封驳太多，效率太低" → 实际上，封驳能避免后期返工，总体效率更高
- **误区2**: "不封驳，快速放行" → 执行阶段发现问题，返工成本是封驳的10倍
- **误区3**: "追求100分方案" → 80分可执行的方案优于100分纸上谈兵

**平衡策略**:
```
第1次审议: 严格标准（必须75分以上）
第2次审议: 适度放宽（80分可放行）
第3次审议: 倾向放行或裁决（85分必须放行）
```

**效率指标**（目标值）:
- 平均封驳次数: ≤ 1.5 次/任务
- 方案起草时长: ≤ 30 分钟
- 审议时长: ≤ 15 分钟
- 端到端完成时长: ≤ 2 小时（简单任务）

**质量指标**（目标值）:
- 批准方案的执行成功率: ≥ 85%
- 执行阶段发现方案缺陷率: ≤ 10%
- 用户满意度: ≥ 90%

| 风险 | 严重程度 | 概率 | 缓解措施 |
|------|----------|------|----------|
| OSS 上传失败 | 高 | 低 | 捕获异常，返回友好提示 |
| 恶意文件上传 | 高 | 中 | 严格校验文件类型和大小 |
| OSS 费用超支 | 中 | 低 | 设置费用告警 |

## 五、回滚方案
**触发条件**: 上传失败率 > 10%
**回滚步骤**: 关闭上传入口，回退代码
**回滚时长**: 5 分钟

## 六、验收标准
- [ ] 支持 JPG、PNG、PDF 上传
- [ ] 文件大小超过 10MB 被拒绝
- [ ] 返回的 URL 可访问
- [ ] 非法文件类型被拒绝
- [ ] 单元测试覆盖率 > 80%

## 七、工时估算
- 开发: 3 小时
- 测试: 1.5 小时
- OSS 配置: 0.5 小时
- **总计**: 5 小时

## 四、进阶技巧

### 4.1 如何写出高质量方案（避免空洞）

**常见空洞方案示例**:
```markdown
## 实施步骤
**步骤1**: 优化系统架构
**步骤2**: 提升性能
**步骤3**: 改善用户体验
```

**问题**:
- 执行者不知道具体做什么
- 无法验收（什么叫"优化完成"？）
- 无法估算工时

**改进后**:
```markdown
## 实施步骤
**步骤1: 优化数据库查询（2小时）**
- 操作: 为 users.phone 字段添加索引
- 命令: `CREATE INDEX idx_phone ON users(phone);`
- 验收: EXPLAIN 显示使用索引，查询时间 < 100ms

**步骤2: 前端懒加载优化（1.5小时）**
- 操作: 图片列表改为滚动加载
- 技术: Intersection Observer API
- 验收: 首屏加载时间 < 2秒

**步骤3: 添加缓存层（1小时）**
- 操作: 热点数据写入 Redis
- 逻辑: 查询前先查 Redis，未命中再查数据库
- 验收: 缓存命中率 > 80%
```

**要点**:
- 步骤具体可执行（有命令、有代码）
- 有明确的输入输出
- 有可验证的标准
- 有工时估算

### 4.2 如何精准封驳（避免过度挑剔）

**过度挑剔示例**:
```markdown
【封驳理由】
1. 变量命名不规范（user_id 应改为 userId）
2. 注释格式不统一
3. 没有使用最新的技术栈
```

**问题**:
- 这些都是小瑕疵，不影响执行
- 过度挑剔导致来回封驳，效率低

**精准封驳示例**:
```markdown
【封驳理由】
1. 缺少数据迁移方案（严重程度: major）
   - 影响: 修改数据库结构，但未说明如何迁移现有数据
   - 建议: 补充迁移 SQL 和回滚脚本

2. API 超时未处理（严重程度: major）
   - 影响: 第三方 API 挂了会导致整个流程卡住
   - 建议: 设置 5秒超时，超时返回友好提示
```

**要点**:
- 只封驳影响执行或风险控制的问题
- 说明影响范围和严重程度
- 给出具体的修改建议
- 小瑕疵可以放行（如注释格式）

### 4.3 如何高效拆解任务（尚书省）

**需求**: 实现用户系统（注册、登录、个人信息管理）。

**方案步骤**:
```markdown
1. 创建数据库表
2. 实现注册 API
3. 实现登录 API
4. 实现个人信息 API
5. 编写单元测试
6. 编写集成测试
7. 更新文档
```

**拆解原则**:
1. 识别依赖关系（表 → API → 测试 → 文档）
2. 按专业分工（工部做API，兵部做测试，礼部做文档）
3. 并行任务优先（注册和登录 API 可并行）

**拆解结果**:
```python
subtasks = [
    {"id": "SUB-1", "type": "implementation", "title": "创建 users 表", "dept": "gongbu", "depends_on": []},
    {"id": "SUB-2", "type": "implementation", "title": "实现注册 API", "dept": "gongbu", "depends_on": ["SUB-1"]},
    {"id": "SUB-3", "type": "implementation", "title": "实现登录 API", "dept": "gongbu", "depends_on": ["SUB-1"]},
    {"id": "SUB-4", "type": "implementation", "title": "实现个人信息 API", "dept": "gongbu", "depends_on": ["SUB-1"]},
    {"id": "SUB-5", "type": "testing", "title": "单元测试", "dept": "bingbu", "depends_on": ["SUB-2", "SUB-3", "SUB-4"]},
    {"id": "SUB-6", "type": "testing", "title": "集成测试", "dept": "bingbu", "depends_on": ["SUB-5"]},
    {"id": "SUB-7", "type": "documentation", "title": "更新 API 文档", "dept": "libu1", "depends_on": ["SUB-2", "SUB-3", "SUB-4"]}
]
```

**执行顺序**:
```

第3批: SUB-5（串行）
第4批: SUB-6（串行）
```

**要点**:
- 明确依赖，避免阻塞
- 最大化并行，缩短总时长
- 分配给合适的部门
