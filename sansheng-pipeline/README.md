# 三省审议流水线 (Sansheng Pipeline)

> 借鉴唐朝三省六部制度，为 Claude Code 构建的多 Agent 协作审议流水线

**版本**: v0.1.0
**状态**: Beta
**作者**: liweizhao

---

## 🎯 这是什么

一个基于**制度化审核机制**的 AI Agent 协作框架，模仿唐朝三省六部制：

```
用户下达任务
  ↓
司礼监(接旨、协调)
  ↓
中书省(起草方案) ⟷ 门下省(审议、封驳)  ←── 自动循环，可封驳最多5次
  ↓ (准奏)
用户最终确认
  ↓
尚书省(派发执行, v0.2)
```

**核心价值**：
- ✅ **强制审核**：不是"可选的 review"，而是**必经环节**
- ✅ **自动循环**：中书省 ↔ 门下省自动迭代，无需人工干预
- ✅ **渐进式质量检查**：5 层检查维度（风险、验收、技术、性能、安全）
- ✅ **可封驳**：门下省有权驳回不合格方案（最多 5 次）
- ✅ **升级机制**：争议超过 5 轮自动上报用户裁决
- ✅ **完整审计**：所有版本、封驳记录可追溯（JSONL 格式）

## 📐 架构设计

### Agent 角色

| Agent | 职责 | 关键能力 |
|-------|------|---------|
| **司礼监** (sililijian) | 流程协调者 | 接旨、协调三省、处理升级、回报 |
| **中书省** (zhongshu) | 方案规划者 | 起草方案、响应封驳修改 |
| **门下省** (menxia) | 方案审议者 | 审议质量、封驳不合格、控制轮次 |
| **尚书省** (shangshu) | 任务派发者（预留） | 拆解任务、派发执行、汇总成果 |

### 通信模式

**Peer-to-Peer**（点对点）：
```
中书省 ⇄ 门下省  (直接 handoff)
门下省 → 司礼监  (准奏/升级时)
司礼监 → 用户    (AskUserQuestion)
```

### 状态管理

所有任务状态存储在：
```
~/.claude/plugins/sansheng-pipeline/data/tasks.json
```

**状态机**：
```
created → planning → reviewing → [rejected ⇄ planning] → approved → done
                                     ↓ (第5次封驳)
                                  escalated → approved
```

**自动循环**：
- 中书省完成方案 → 自动 handoff 门下省审议
- 门下省封驳 → 自动 handoff 中书省修改 → 回到审议
- 门下省准奏 → 返回用户确认

## 🎯 渐进式质量检查

**设计理念**：门下省不一次性抛出所有问题，而是每次封驳只提**一个维度**的要求，让中书省逐步完善方案。

### 检查维度（5 层）

| 审议轮次 | 检查维度 | 检查内容 | 封驳时中书省修改 |
|---------|---------|---------|----------------|
| 第 0 次 | 基础形式 + 风险评估 | 长度、结构、目标、风险与应对 | 补充风险评估 |
| 第 1 次 | 验收标准 | 功能验收、性能基准、安全清单 | 补充验收标准 |
| 第 2 次 | 技术细节 | 技术栈选型、架构设计、实施细节 | 补充技术细节 |
| 第 3 次 | 性能指标 | 响应时间、吞吐量、资源消耗 | 补充性能指标 |
| 第 4 次 | 安全审计 | 权限控制、数据加密、审计日志 | 补充安全审计 |

### 示例流程

**任务**：实现用户注册系统

**第 0 次审议**（检查风险评估）：
- 中书省 v1：基础方案（有目标、步骤，但缺少风险评估）
- 门下省封驳：❌ 缺少风险评估章节
- 中书省 v2：补充"技术风险、集成风险、时间风险"

**第 1 次审议**（检查验收标准）：
- 中书省 v2：已有风险评估，但缺少验收标准
- 门下省封驳：❌ 缺少验收标准章节
- 中书省 v3：补充"功能验收、性能指标、代码质量"

**第 2 次审议**（检查技术细节）：
- 中书省 v3：已有验收标准，但缺少技术细节
- 门下省封驳：❌ 缺少技术细节说明
- 中书省 v4：补充"技术栈选型、架构设计、实施细节"

**第 3 次审议**（检查性能指标）：
- 中书省 v4：已有技术细节，但缺少性能指标
- 门下省封驳：❌ 缺少性能指标章节
- 中书省 v5：补充"响应时间 P95 < 200ms、QPS > 1000"

**第 4 次审议**（检查安全审计）：
- 中书省 v5：已有性能指标，但缺少安全审计
- 门下省封驳：❌ 缺少安全审计章节
- 中书省 v6：补充"权限控制、数据加密、审计日志"

**第 5 次审议**（全面检查）：
- 中书省 v6：已补充所有维度
- 门下省准奏：✅ 方案完整，可以执行

**如果第 5 次仍封驳**：自动升级司礼监裁决

## 🔧 技术实现

### 核心库

**`lib/task_state.py`**：
- 任务创建、查询、更新
- 版本历史管理（每次规划是一个版本）
- 封驳记录追踪
- 原子操作（文件锁）

**关键 API**：
```python
from task_state import (
    create_task,           # 创建任务
    add_plan_version,      # 添加方案版本
    add_rejection,         # 记录封驳
    set_escalation,        # 升级裁决
    set_approval,          # 用户批准
    get_rejection_count    # 查询封驳次数
)
```

### Agent SOUL.md

每个 agent 的人格、职责、工作流都定义在 `agents/{id}/SOUL.md`：

- **司礼监**：5 个阶段流程 + 异常处理
- **中书省**：方案模板 + 封驳响应机制
- **门下省**：5 维度审查标准 + 封驳/准奏/升级决策树

### Handoff 机制

**`mcp-server/src/handoff.ts`**：
- 消息验证：检查 JSON Schema 格式
- 审计日志：记录所有 handoff 事件
- 重试机制：指数退避（5s, 15s, 45s），最多 3 次
- 自动升级：失败后上报司礼监

**Handoff 消息格式**：
```json
{
  "task_id": "TASK-20260311-001",
  "from_agent": "zhongshu",
  "to_agent": "menxia",
  "action": "review",
  "content": { "version": 1 },
  "timestamp": "2026-03-11T10:30:45.123Z"
}
```

### 审计日志

**存储位置**：`~/.claude/plugins/sansheng-pipeline/data/audit/audit-YYYYMMDD.jsonl`

**关键事件**：
- `handoff_initiated`：发起 handoff
- `handoff_completed`：handoff 成功
- `handoff_failed`：handoff 失败
- `plan_submitted`：中书省提交方案
- `plan_approved`：门下省准奏
- `plan_rejected`：门下省封驳

**查询示例**：
```bash
# 查看今日所有 handoff 事件
grep 'handoff' ~/.claude/plugins/sansheng-pipeline/data/audit/audit-$(date +%Y%m%d).jsonl
```

## 📊 使用示例

### 场景 1: 数据库架构调整

**任务**：给 users 表增加 role 字段，支持多角色用户

**流程**：
1. 司礼监收集信息（表结构、用户量）
2. 中书省起草方案：
   ```
   1. 迁移脚本：ALTER TABLE users ADD role VARCHAR(50)
   2. 数据回填：UPDATE users SET role = 'user' WHERE role IS NULL
   3. 权限适配：修改认证中间件
   4. 测试：单元测试 + 灰度验证
   ```
3. 门下省审议：✅ 准奏
4. 用户确认：批准
5. 输出：可执行方案

**耗时**：~10 分钟

---

### 场景 2: 方案封驳修改

**任务**：优化用户列表接口性能

**流程**：
1. 中书省方案 v1：增加 Redis 缓存
2. 门下省封驳：**缺少缓存失效策略**
3. 中书省方案 v2：补充 TTL 60s + 更新时主动失效
4. 门下省准奏：✅
5. 用户确认：批准

**耗时**：~15 分钟（1 次封驳）

---

### 场景 3: 渐进式质量提升

**任务**：实现 API 限流功能

**流程**（自动循环审议）：
1. 中书省方案 v1：基础方案（有目标、步骤）
2. 门下省封驳 1：❌ 缺少风险评估
3. 中书省方案 v2：补充风险评估
4. 门下省封驳 2：❌ 缺少验收标准
5. 中书省方案 v3：补充验收标准
6. 门下省封驳 3：❌ 缺少技术细节
7. 中书省方案 v4：补充技术细节（Redis + Lua 脚本）
8. 门下省封驳 4：❌ 缺少性能指标
9. 中书省方案 v5：补充性能指标（QPS > 10000）
10. 门下省准奏：✅
11. 用户确认：批准

**耗时**：~20 分钟（4 次封驳，全自动）

---

### 场景 4: 升级用户裁决

**任务**：从 session 改为 JWT 认证

**流程**：
1. 中书省方案 v1：全面切换
2. 门下省封驳 1：❌ 缺少风险评估
3. 中书省方案 v2：补充风险评估
4. 门下省封驳 2：❌ 缺少验收标准
5. 中书省方案 v3：补充验收标准
6. 门下省封驳 3：❌ 未考虑旧用户迁移（技术细节问题）
7. 中书省方案 v4：30 天兼容期
8. 门下省封驳 4：❌ 建议 7 天兼容期（意见分歧）
9. 中书省方案 v5：坚持 30 天（附详细理由）
10. 门下省封驳 5：❌ 仍建议 7 天
11. **封驳 5 次 → 自动升级司礼监裁决**
12. 司礼监整理争议点：7 天 vs 30 天（性能 vs 用户体验）
13. 用户裁决：14 天折中方案
14. 输出：14 天兼容期方案

**耗时**：~30 分钟（5 次封驳 + 1 次裁决）

## 📚 查询命令

### 列出所有任务

```bash
cd ~/.claude/plugins/sansheng-pipeline
python3 -c "
import sys; sys.path.insert(0, 'lib')
from task_state import list_tasks
for t in list_tasks():
    print(f\"{t['id']}: [{t['state']}] {t['title']}\")
"
```

### 查看任务详情

```bash
python3 -c "
import sys; sys.path.insert(0, 'lib')
from task_state import get_task
import json
task = get_task('TASK-20260310-001')
print(json.dumps(task, ensure_ascii=False, indent=2))
"
```

### 查看封驳历史

```bash
python3 -c "
import sys; sys.path.insert(0, 'lib')
from task_state import get_task
task = get_task('TASK-20260310-001')
for i, r in enumerate(task['rejections'], 1):
    print(f'{i}. {r[\"timestamp\"]}: {r[\"reason\"][:80]}...')
"
```
