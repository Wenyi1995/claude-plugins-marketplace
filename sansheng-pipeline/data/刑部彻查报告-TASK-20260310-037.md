## 问题清单

### P0（严重，阻塞自动化）

#### 1. **MCP Server 无自动流转机制**
- **现象**: `sansheng_review_all` 执行完后状态停留在 `approved`，需要用户手动调用 `sansheng_finalize` 才能完成
- **根因**:
  - `sansheng.ts:177` 准奏后只返回结果，不自动更新为 `done` 状态
  - 司礼监 SOUL.md 设计为"请求用户确认" → 手动调用 finalize
  - 违背了"自动批准"的设计目标
- **影响**:
  - 门下省准奏后流程中断
  - 需要圣上介入每个任务的批准（违背自动化原则）
  - 36 个任务中 17 个停在 `approved` 状态
- **证据**:
  ```typescript
  // sansheng.ts:177-188
  if (reviewResult.decision === 'approved') {
    taskState.updateState(taskId, 'approved', `门下省准奏方案 v${version}`);
    return {
      status: 'approved',
      task_id: taskId,
      summary: '方案已通过三省审议，请确认是否批准执行',  // ← 需要确认
      final_plan: currentPlan,
    };
  }
  ```
- **修复建议**:
  - 方案 A（推荐）: `sansheng_review_all` 准奏后自动调用内部 finalize 逻辑，直接更新为 `done`
  - 方案 B: 增加参数 `auto_approve: boolean`，默认 true

---

#### 2. **三省没有独立的 handoff 工具，依赖 Claude Code 内置能力**
- **现象**:
  - agents/*/SOUL.md 中使用 `@zhongshu`、`@menxia` 语法
  - 这是 Claude Code 内置的 agent 切换功能，不是三省自己的工具
  - `lib/handoff_utils.py` 和 `lib/handoff_validator.py` 只是测试占位符
- **根因**:
  - `handoff_utils.py:19-41` 的 `agent_handoff` 只是 mock 函数（90% 成功率模拟）
  - 实际 handoff 依赖 Claude Code 的 `@agent` 语法
  - 没有真正的 MCP tool 实现 handoff
- **影响**:
  - handoff 记录无法进入审计日志
  - 无法实现重试、超时、失败升级机制
  - 依赖外部系统（Claude Code），非独立运作
- **证据**:
  ```python
  # lib/handoff_utils.py:34-41
  # 模拟：90% 成功率
  import random
  if random.random() < 0.9:
      print(f"[HANDOFF] {from_agent} -> {to_agent}: 成功")
      return HandoffResult(success=True, message="Handoff successful")
  else:
      print(f"[HANDOFF] {from_agent} -> {to_agent}: 失败")
      raise Exception("Handoff failed: network timeout")
  ```
- **修复建议**:
  - 将 handoff 功能作为 MCP tool 暴露：`sansheng_handoff(from, to, message)`
  - 内部调用 Claude Code 的 agent 切换 API（如有）
  - 记录所有 handoff 到审计日志

#### 3. **MCP Server 绕过了 Agent SOUL.md 的完整流程**
- **现象**:
  - `sansheng_review_all` 内部直接调用 `zhongshuPlan()` 和 `menxiaReview()` 函数
  - 中书省和门下省 agent 的 SOUL.md 被完全跳过
  - agent 的审议逻辑、状态记录、handoff 规范全部失效
- **根因**:
  - 设计上为了"简化流程"，把中书省和门下省的逻辑用 TypeScript 重写
  - `sansheng.ts:46-151` 实现了简化版的起草和审议
  - 违背了"agent 自治"原则
- **影响**:
  - agent SOUL.md 中定义的规范（如"封驳理由必须具体"）无法生效
  - 无法利用 agent 的推理能力（如 oracle 复杂审议）
  - 中书省和门下省的培训文档失效
- **证据**:
  ```typescript
  // sansheng.ts:156-188
  export async function sanshengReviewAll(args: ReviewAllArgs): Promise<ReviewAllResult> {
    // 直接调用内部函数，跳过 agent
    let currentPlan = zhongshuPlan(taskId, context, version);
    const reviewResult = menxiaReview(taskId, currentPlan, version, rejectionCount);
  }
  ```
- **修复建议**:
  - 方案 A（彻底重构）: MCP Server 只负责协调，真正的起草/审议由 agent 执行
  - 方案 B（渐进改进）: 保留 MCP 简化流程，但增加"完整流程"模式开关

### P1（重要，影响体验）

#### 4. **封驳逻辑过于机械，不符合真实审议**
- **现象**:
  - 第 1 次审议固定检查"风险评估"
  - 第 2 次审议固定检查"验收标准"
  - 第 3 次审议固定检查"技术细节说明"
  - 无法根据方案实际内容灵活审议
- **根因**:
  - `sansheng.ts:126-147` 硬编码了审议规则
  - 没有真正的"质量评估"逻辑
- **影响**:
  - 好方案可能被机械封驳（因为缺少某个章节）
  - 差方案可能通过审议（因为有章节但内容空洞）
- **证据**:
  ```typescript
  // sansheng.ts:126-147
  if (currentRejectionCount === 0) {
    if (!hasRiskAssessment) {
      const reason = `方案 v${version} 缺少风险评估章节...`;
      taskState.addRejection(taskId, reason, 'menxia');
      return { decision: 'rejected', reason };
    }
  }
  ```
- **修复建议**:
  - 引入 LLM 审议（调用 oracle agent）
  - 基于"必须达标"清单（门下省 SOUL.md:316-326）动态检查

#### 5. **36 个任务状态异常，无自动清理机制**

- **影响**:

  - 增加状态机完整性检查：检测长期停滞的任务

## 根因分析

### 架构层面
1. **设计冲突**: 圣上期望"自动化"，但司礼监 SOUL.md 设计为"用户确认"
2. **职责混淆**: MCP Server 既做协调又做执行，侵蚀了 agent 的职责边界
3. **依赖外部**: handoff 依赖 Claude Code 内置能力，无法独立运作

### 实现层面
1. **简化过度**: 为了"快速验证"，跳过了 agent 完整流程
2. **状态不一致**: `approved` 应该自动流转到 `done`，但需要手动触发
3. **缺少闭环**: 准奏后没有自动执行，escalated 后没有自动裁决

### 配置层面
1. **SOUL.md 与代码脱节**: agent 文档定义的流程与实际执行不一致
2. **缺少审计**: handoff 没有统一的记录机制
3. **缺少监控**: 没有任务状态健康度检查
